## 개요

리소스 마켓플레이스는 플러그인 생성 시 재사용 가능한 공유 자원(가이드, 템플릿, 샘플, 도구)의
카탈로그이며, 필요한 리소스를 플러그인 안으로 가져가 사용하기 위한 공유 자원 풀임.

**리소스 유형 요약:**

| 유형 | 설명 | 디렉토리 |
|------|------|---------|
| 가이드 | 플러그인 제작에 공통으로 필요한 참조 문서 | `resources/guides/{2차분류}/` |
| 템플릿 | 표준 파일의 스켈레톤/보일러플레이트 | `resources/templates/{2차분류}/` |
| 샘플 | 재사용 가능한 코드 스니펫/패턴 | `resources/samples/{2차분류}/` |
| 도구 | 커스텀 제작/외부 도구의 설치·사용법 카탈로그 | `resources/tools/` |

**공통 기여 프로세스:**

1. **파일 배치** — 리소스 파일을 해당 디렉토리에 배치
2. **문서 작성** — 도구의 경우 툴 가이드 문서 작성 (가이드의 경우 가이드 문서 자체가 해당)
3. **카탈로그 등록** — `plugin-resources.md`의 해당 목록 테이블에 행 추가

## 리소스 디렉토리 구조

전체 디렉토리 구조:

```
resources/
├── plugin-resources.md          # 리소스 카탈로그 (모든 리소스의 목록)
├── guides/                      # 가이드 문서
│   ├── dify/                    # 2차 분류: dify
│   └── plugin/                  # 2차 분류: plugin
├── templates/                   # 템플릿 파일
│   ├── dify/
│   ├── general/
│   └── plugin/
├── samples/                     # 샘플 파일
│   └── plugin/
└── tools/                       # 도구
    ├── *.md                     # 툴 가이드 문서 (도구별 1개)
    └── customs/                 # 커스텀 도구 소스
        ├── .env                 # 공통 환경 변수
        ├── dify-cli/            # 2차 분류: dify-cli
        ├── diagram/             # 2차 분류: diagram
        └── general/             # 2차 분류: general
```

**2차 분류 규칙:**

| 항목 | 규칙 |
|------|------|
| 명명 | kebab-case (예: `dify-cli`, `general`) |
| 기준 | 도메인 또는 용도 기반 |
| 생성 | 기존 분류에 해당하지 않으면 새 서브디렉토리 생성 |

> 새 2차 분류 추가 시 디렉토리만 생성하면 됨.
> 별도의 분류 선언이나 등록 절차는 불필요.

## 가이드 추가

가이드 추가는 3단계로 진행.

### 단계 1. 파일 배치

가이드 파일을 다음 경로에 배치:

```
resources/guides/{2차분류}/{guide-name}.md
```

- 파일명: kebab-case 사용 (예: `plugin-dev-guide.md`)
- 2차 분류 디렉토리가 없으면 생성

### 단계 2. 가이드 문서 작성

마크다운 작성 규칙 준수 필수 (상세 규칙은 [부록 A](#부록-a-마크다운-작성-규칙) 참조):

| 규칙 | 설명 |
|------|------|
| 명사체 | 명사형 종결어미 사용 ("~함", "~임", "~됨") |
| 줄바꿈 | 한 줄 120자 이내 |
| TOC | 첫 H1 아래에 GitHub 앵커 스타일 목차 |
| 구분선 | 모든 H2 위에 `---` 수평선 |
| Top 링크 | 각 H2 섹션 마지막에 `[Top](#제목-앵커)` |

> **참고:** [부록 A](#부록-a-마크다운-작성-규칙)의 내용을 복사하여
> 사용 중인 런타임 AI의 지시 파일(`CLAUDE.md`, `AGENTS.md` 등)에 붙여넣으면
> AI가 자동으로 마크다운 작성 규칙을 준수함.

### 단계 3. 카탈로그 등록

[부록 B](#부록-b-카탈로그-등록-프롬프트)의 프롬프트를 AI에게 전달하여 자동 등록.

> **참고 — 컬럼 구조:** `2차 분류 | 가이드명 | 설명 | 상세`
>
> ```markdown
> | plugin | my-guide | 플러그인 작성 시 참고할 가이드 | [상세](guides/plugin/my-guide.md) |
> ```

## 커스텀 도구 추가

커스텀 도구는 4가지 카테고리로 분류:

| 카테고리 | 설명 | 소스 배치 | 예시 |
|---------|------|----------|------|
| 커스텀 앱 | Python/Node.js 등으로 구현한 프로그램 | `customs/{분류}/` | generate_image |
| 커스텀 CLI | 셸 스크립트로 구현한 도구 | `customs/{분류}/` | check-mermaid |
| MCP 서버 | Model Context Protocol 기반 외부 패키지 | 불필요 (외부) | context7 |
| LSP 서버 | Language Server Protocol 기반 코드 분석 | 불필요 (외부) | python-lsp-server |

> **LSP 서버 참고:** LSP 서버는 `install.yaml`의 `lsp_servers` 항목에서 직접 선언하며,
> 별도의 툴 가이드 문서나 카탈로그 등록이 불필요함.

**공통 3단계 프로세스:**

1. **소스 배치** — 커스텀 앱/CLI만 해당 (MCP 서버는 외부 패키지이므로 불필요)
2. **툴 가이드 작성** — `resources/tools/{tool-name}.md`에 도구별 가이드 문서 작성
3. **카탈로그 등록** — `plugin-resources.md`의 "도구 목록" 테이블에 행 추가

**카탈로그 도구 목록 컬럼 구조:** `카테고리 | 도구명 | 설명 | 상세`

### 커스텀 앱 추가

커스텀 앱은 Python, Node.js 등으로 구현한 독립 실행 프로그램임.
소스 배치, 툴 가이드 작성, 카탈로그 등록 3단계로 진행.

#### 단계 1. 소스 배치

소스 파일을 다음 경로에 배치:

```
resources/tools/customs/{분류}/{파일명}
```

**배치 규칙:**

| 항목 | 규칙 |
|------|------|
| 소스 파일 | `customs/{분류}/` 아래에 배치 (예: `customs/general/generate_image.py`) |
| 의존성 파일 | `requirements.txt` (Python) 또는 `package.json` (Node.js) 포함 권장 |
| 환경 변수 예시 | `.env.example` 파일 포함 권장 |
| 분류 디렉토리 | 기존 분류에 해당하지 않으면 새 디렉토리 생성 |

**디렉토리 배치 예시:**

```
resources/tools/customs/general/
├── generate_image.py          # 소스 파일
├── requirements.txt           # 의존성 목록 (권장)
└── .env.example               # 환경 변수 예시 (권장)
```

#### 단계 2. 툴 가이드 작성

`resources/tools/{tool-name}.md` 경로에 툴 가이드 문서 작성.
파일명은 kebab-case 사용 (예: `generate-image.md`).

**커스텀 앱 툴 가이드 템플릿:**

```markdown

  - [설치 정보](#설치-정보)
  - [환경 변수](#환경-변수)
  - [명령어](#명령어)

### 커스텀 CLI 추가

커스텀 CLI는 셸 스크립트로 구현한 경량 도구임.
커스텀 앱과 동일한 3단계이나 툴 가이드 구조가 다름.

#### 단계 1. 소스 배치

셸 스크립트를 다음 경로에 배치:

```
resources/tools/customs/{분류}/{script-name}.sh
```

- 실행 권한 설정: `chmod +x {script-name}.sh`
- Windows 환경은 `.bat` 또는 `.cmd` 확장자 사용

#### 단계 2. 툴 가이드 작성

커스텀 CLI의 툴 가이드는 커스텀 앱과 구조가 다름.
환경 변수, 명령어 섹션이 없고 컨테이너 실행 가이드가 포함될 수 있음.

**커스텀 CLI 툴 가이드 템플릿:**

```markdown
# {tool-name}

- [{tool-name}](#{tool-name-앵커})
  - [기본 정보](#기본-정보)
  - [컨테이너 실행 가이드](#컨테이너-실행-가이드)

## 부록 A. 마크다운 작성 규칙

아래 내용을 사용 중인 런타임 AI의 지시 파일에 복사하여 붙여넣기하면
AI가 마크다운 작성 규칙을 자동으로 준수함.

- **Claude Code** → `CLAUDE.md`에 붙여넣기
- **Codex CLI** → `AGENTS.md`에 붙여넣기
- **Gemini CLI** → `GEMINI.md`에 붙여넣기

**복사 범위:** 아래 코드블록 전체를 복사하여 사용.

````markdown
# 마크다운 작성 가이드

## 기본
- 문서 작성 시 명사체(명사형 종결어미) 사용
  - 예시: "~한다" → "~함", "~이다" → "~임", "~된다" → "~됨"
  - 예시: "지원한다" → "지원", "사용할 수 있다" → "사용 가능"
- 한 줄은 120자 이내로 작성, 긴 문장은 적절히 줄바꿈
  - 마크다운 줄바꿈: 문장 끝에 스페이스 2개 + 줄바꿈
- 간결하고 객관적인 기술 문서 스타일 유지

## 목차 및 구분선

### 1. 목차(TOC) 추가
- 파일 최상단의 `# 제목` 바로 아래에 목차를 추가
- 목차는 `# `, `## `, `### ` 헤딩을 기반으로 생성
- 형식: 마크다운 링크 목록 (GitHub 스타일 앵커)
- 첫 번째 `# ` 항목은 `- [제목](#앵커)` 형식 (인덴트 없음, 최상위)
- 두 번째 이후 `# ` 항목은 `- [제목](#앵커)` 형식 (인덴트 없음)
- `## ` 항목은 `  - [제목](#앵커)` 형식 (인덴트 2칸)
- `### ` 항목은 `    - [제목](#앵커)` 형식 (인덴트 4칸)

**앵커 생성 규칙:**
- 소문자 변환
- 공백 → `-`
- 특수문자(괄호, 마침표, 콤마, 콜론 등) 제거
- 한글은 그대로 유지
- 예: `## 1.1 개요` → `#11-개요`
- 예: `### 단계 1. 문서 준비` → `#step-1-문서-준비`

### 2. `---` 구분선 추가
- 모든 `## ` (level-2) 헤딩 바로 위에 `---` 수평선 추가

### 3. `[Top]` 링크 추가
- 각 `## ` (level-2) 섹션의 마지막에 `[Top](#제목-앵커)` 링크 추가
- 앵커는 파일의 `# 제목`에 해당하는 앵커
- 다음 `---` 바로 위에 위치
- 파일의 마지막 `## ` 섹션에도 끝에 `[Top]` 링크 추가

### 주의사항
- 기존 내용은 절대 수정하지 않음
- 이미 `---`나 `[Top]`이 있는 파일은 중복 추가하지 않음
- `# ` (level-1) 헤딩이 파일에 여러 개 있을 수 있음
  - 목차는 첫 번째 `# ` 헤딩 기준으로 생성
  - `[Top]` 링크는 `## ` (level-2) 섹션에만 추가
  - `# ` (level-1) 헤딩에는 `[Top]` 링크를 추가하지 않음
- 코드블록(``` 또는 ~~~) 안의 `##`는 헤딩이 아니므로 무시
````
