## 환영합니다

DMAP 프로젝트에 관심을 가져주셔서 감사함.
본 프로젝트는 코드 없이 Markdown과 YAML만으로 멀티에이전트 플러그인을 만드는
선언형 프레임워크임.

누구나 기여 가능하며, 버그 리포트부터 새로운 리소스 추가까지 다양한 방식으로 참여 가능함.

## 시작하기

### 사전 요구사항

- Git 설치
- GitHub 계정
- 텍스트 에디터 (Markdown 편집 가능)

### Fork & Clone

```bash
# 1. GitHub에서 저장소 Fork (웹 UI에서 진행)

# 2. Fork한 저장소 Clone
git clone https://github.com/{your-username}/gen-dmap.git
cd gen-dmap

# 3. 원본 저장소를 upstream으로 추가
git remote add upstream https://github.com/{원본소유자}/gen-dmap.git
```

### 브랜치 생성

```bash
# 최신 main 동기화
git fetch upstream
git checkout main
git merge upstream/main

# 작업 브랜치 생성
git checkout -b {브랜치유형}/{설명}
```

브랜치 명명 규칙:

| 유형 | 접두사 | 예시 |
|------|--------|------|
| 기능 추가 | `feature/` | `feature/add-python-template` |
| 버그 수정 | `fix/` | `fix/typo-in-agent-standard` |
| 문서 개선 | `docs/` | `docs/improve-gateway-guide` |
| 리소스 추가 | `resource/` | `resource/add-mermaid-tool` |

## 기여 워크플로우

1. **Fork & Clone** — GitHub에서 저장소를 Fork하고 로컬에 Clone
2. **브랜치 생성** — 작업 유형에 맞는 브랜치 생성 (예: `feature/add-dify-guide`)
3. **변경 작업** — 파일 추가, 수정, 삭제 등 필요한 작업 수행
4. **커밋** — 변경사항을 논리적 단위로 커밋 (한글 커밋 메시지)
   ```bash
   git add {변경 파일}
   git commit -m "리소스 카탈로그에 Dify 가이드 추가"
   ```
5. **Push** — 작업 브랜치를 원격 저장소(Fork)에 Push
   ```bash
   git push origin feature/add-dify-guide
   ```
6. **Pull Request 생성** — GitHub 웹 UI에서 원본 저장소의 `main` 브랜치로 PR 생성

## 리소스 마켓플레이스 기여

리소스 마켓플레이스는 DMAP 플러그인 제작에 필요한 공유 자원 풀임.
상세 가이드는 [리소스 기여 가이드](resources/guides/plugin/resource-contribution-guide.md)를 참조.

### 리소스 유형

| 유형 | 설명 | 디렉토리 |
|------|------|---------|
| 가이드 | 플러그인 제작에 공통으로 필요한 참조 문서 | `resources/guides/{분류}/` |
| 템플릿 | 표준 파일의 스켈레톤/보일러플레이트 | `resources/templates/{분류}/` |
| 샘플 | 재사용 가능한 코드 스니펫/패턴 | `resources/samples/{분류}/` |
| 도구 | 커스텀 제작/외부 도구의 설치·사용법 카탈로그 | `resources/tools/` |

### 공통 기여 프로세스

1. **파일 배치** — 리소스 파일을 적절한 디렉토리에 배치
   - 가이드 → `resources/guides/{분류}/`
   - 템플릿 → `resources/templates/{분류}/`
   - 샘플 → `resources/samples/{분류}/`
   - 도구 가이드 → `resources/tools/`, 소스 → `resources/tools/customs/{분류}/`
2. **문서 작성** — 마크다운 작성 규칙 준수 (도구의 경우 툴 가이드 문서 작성)
3. **카탈로그 등록** — `resources/plugin-resources.md`의 해당 목록 테이블에 행 추가

> **참고**: 상세 절차, 파일명 규칙, 템플릿은
> [리소스 기여 가이드](resources/guides/plugin/resource-contribution-guide.md) 참조

## 표준 문서 기여

`standards/` 디렉토리의 문서는 DMAP의 핵심 스펙이므로 신중한 변경 필요.

- **논의 필수** — 반드시 Issue에서 변경 제안 후 합의 도출
- **하위 호환성** — 기존 플러그인에 영향을 주지 않도록 하위 호환성 유지
- **마크다운 규칙 미적용** — 표준 문서는 자체 포맷을 따르며,
  일반 마크다운 작성 규칙(명사체, TOC, 구분선, Top 링크)을 적용하지 않음

[Top](#dmap-기여-가이드)

## 마크다운 작성 규칙

> **주의**: 본 규칙은 `standards/` 디렉토리 문서에는 적용하지 않음.
> `resources/` 및 기타 일반 문서에만 적용.

| 규칙 | 설명 |
|------|------|
| 명사체 사용 | "~함", "~임", "~됨" 등 명사형 종결어미 사용 |
| 줄바꿈 | 한 줄 120자 이내로 작성 |
| TOC 추가 | 파일 최상단 H1 바로 아래에 GitHub 앵커 스타일 목차 추가 |
| 구분선 | 모든 H2 위에 `---` 수평선 추가 |
| Top 링크 | 각 H2 섹션 마지막에 `[Top](#앵커)` 링크 추가 |

**앵커 생성 규칙:**

- 소문자 변환
- 공백 → `-`
- 특수문자(괄호, 마침표, 콤마, 콜론 등) 제거
- 한글은 그대로 유지
- 예: `## 1.1 개요` → `#11-개요`

## 커밋 메시지 규칙

- 커밋 메시지는 **한글**로 작성
- 간결하고 명확하게 변경 내용 기술
- 예시:

  - `리소스 카탈로그에 generate_image 도구 등록`
  - `플러그인 개발 가이드 오탈자 수정`

## Pull Request 가이드

### PR 작성 요령

- **제목** — 변경 내용을 한 줄로 요약 (한글)
- **본문** — 다음 항목 포함:
  - 변경 이유 (Why)
  - 변경 내용 (What)
  - 테스트 방법 (How to verify)
- **Issue 연결** — 관련 Issue가 있으면 `Resolves #123` 형태로 연결

### PR 체크리스트

- [ ] 마크다운 작성 규칙 준수 (표준 문서 제외)
- [ ] 리소스 추가 시 카탈로그(`resources/plugin-resources.md`) 등록 완료
- [ ] 기존 문서/기능에 영향이 없는지 확인
- [ ] 커밋 메시지가 한글로 작성되었는지 확인
- [ ] 테스트 또는 검증 수행 (해당 시)

## Issue 가이드

### 버그 리포트

필수 정보:

- **환경** — OS, 런타임 버전 (예: Windows 11, Claude Code v1.x)
- **재현 단계** — 버그를 재현할 수 있는 구체적 절차
- **기대 결과** — 예상했던 동작
- **실제 결과** — 실제 발생한 동작
- **추가 정보** — 스크린샷, 로그, 에러 메시지 등

### 기능 제안

필수 정보:

- **해결하려는 문제** — 현재 불편한 점이나 필요성
- **제안하는 해결 방안** — 구체적인 기능 제안
- **대안** — 다른 해결 방법이 있다면 함께 제시
- **추가 정보** — 참고 자료, 유사 사례 등

## 행동 강령

본 프로젝트는 개방적이고 환영하는 환경을 유지하기 위해 다음 원칙을 준수함:

- **상호 존중** — 모든 기여자를 존중하며, 다양한 관점과 경험을 환영
- **건설적인 피드백** — 비판은 긍정적이고 건설적인 방식으로 제공
- **다양성 존중** — 나이, 장애, 민족, 성별, 경험 수준, 국적, 인종, 종교 등에
  관계없이 모든 사람을 환영
- **개인 공격 금지** — 개인에 대한 공격, 모욕, 비하 발언 금지

위반 시 프로젝트 관리자는 기여자를 프로젝트에서 일시적 또는 영구적으로 제외할 수 있음.
