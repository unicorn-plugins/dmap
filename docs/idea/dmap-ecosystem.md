## 개요

DMAP 생태계는 4개의 계층으로 구성됨.
DMAP 빌더가 AI팀(플러그인)을 생성하고,
AI팀은 Abra를 개발 도구로 사용하여 AI 서비스를 만드는 구조.

서비스 범위는 **AI 서비스**로 한정하고,
개발 언어는 **Python**으로 통일하여 집중과 품질을 확보함.

## 생태계 역할 정의

| 현실 세계 비유 | DMAP 생태계 | 역할 |
|--------------|------------|------|
| 인력파견 회사 | DMAP 빌더 | 멀티에이전트 플러그인 생성기 |
| 프로젝트 투입 팀 | 목적별 플러그인 | AI팀 (도메인 전문가 집단) |
| 팀의 개발 도구 | Abra 플러그인 | AI팀의 서비스 개발 도구 |
| 팀이 납품한 서비스 | AI 서비스 | AI팀이 만든 결과물 |

### 핵심 원칙

- **AI팀은 도메인 지식에 집중** — 서비스 구현은 Abra에 위임
- **Abra는 공통 개발 키트** — 모든 AI팀이 동일한 도구로 서비스 개발
- **서비스 범위: AI 한정** — 범용 서비스가 아닌 AI 서비스에 집중
- **개발 언어: Python 한정** — AI/ML 생태계 표준 언어로 통일

## 서비스 유형

AI팀(플러그인)이 만들 수 있는 서비스는 2가지 유형으로 한정.

| 유형 | 설명 | 특징 | 개발 도구 |
|------|------|------|----------|
| Dify Agent AI 서비스 | Dify 워크플로우 기반 AI 서비스 | 노코드, 빠른 프로토타이핑 | Abra |
| Python AI 서비스 | Python 코드 기반 AI 서비스 | 유연한 커스터마이징 | Abra |

### 범위 한정 근거

| 관점 | 효과 |
|------|------|
| 집중 | "AI 서비스 개발 도구"라는 명확한 정체성 확보 |
| 생태계 | Python = AI/ML 사실상 표준 (LangChain, FastAPI, Dify 등) |
| 품질 | 하나를 잘 하는 것 > 여럿을 어중간하게 하는 것 |
| 학습곡선 | 플러그인 개발자가 Python만 알면 충분 |

## 구체 예시

### 예시 1: blog-poster 플러그인

```
[DMAP 빌더] → blog-poster 플러그인 생성

blog-poster AI팀 구성:
  - 콘텐츠 기획자 에이전트
  - 콘텐츠 작성자 에이전트
  - 편집자 에이전트
  - 배포자 에이전트

서비스 개발 요청:
  "매일 자동으로 블로그 포스팅하는 서비스 만들어줘"
       ↓
  [Abra 플러그인]
       ↓ 5단계 파이프라인
  매일 자동 포스팅 Dify Agent 서비스
```

### 예시 2: gen-ai-curriculum 플러그인

```
[DMAP 빌더] → gen-ai-curriculum 플러그인 생성

gen-ai-curriculum AI팀 구성:
  - 교재 분석 에이전트
  - 커리큘럼 작성 에이전트
  - 프리젠테이션 작성 에이전트

서비스 개발 요청:
  "교재 업로드하면 커리큘럼 자동 생성하는 서비스 만들어줘"
       ↓
  [Abra 플러그인]
       ↓ Python 파이프라인
  교재 분석 → 커리큘럼 생성 Python AI 서비스
```
